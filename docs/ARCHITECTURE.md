## 🎯 설계 원칙

### 1. Domain-Driven Design (DDD)

각 비즈니스 도메인을 독립적인 모듈로 구성:

```
domains/
├── auth/           # 인증 (로그인, 회원가입, 토큰)
├── user/           # 사용자 관리
├── company/        # 회사 설정
├── customer/       # 고객 관리 ⭐ NEW
├── line_item/      # 라인 아이템 ⭐ 구조 변경
├── estimate/       # 견적서
├── invoice/        # 인보이스
└── subscription/   # 구독/결제 ⭐ NEW
```

### 2. Layered Architecture

```
┌─────────────────────────────────────────┐
│              API Layer                   │  ← HTTP 요청/응답 처리
│         (api.py, FastAPI Router)         │
├─────────────────────────────────────────┤
│            Schema Layer                  │  ← 입력 검증, 응답 직렬화
│          (schemas.py, Pydantic)          │
├─────────────────────────────────────────┤
│           Service Layer                  │  ← 비즈니스 로직
│             (service.py)                 │
├─────────────────────────────────────────┤
│          Repository Layer                │  ← 데이터 접근
│           (repository.py)                │
├─────────────────────────────────────────┤
│            Model Layer                   │  ← ORM 엔티티
│        (models.py, SQLAlchemy)           │
└─────────────────────────────────────────┘
```

### 3. Domain 구조

각 도메인은 동일한 구조를 따름:

```
domains/estimate/
├── __init__.py
├── models.py          # SQLAlchemy 모델
├── schemas.py         # Pydantic 스키마 (Request/Response)
├── repository.py      # 데이터 접근 계층
├── service.py         # 비즈니스 로직
└── api.py             # FastAPI 라우터
```

### Token 구조

```python
# Access Token (30분 유효)
{
    "sub": "user_uuid",           # 사용자 ID
    "company_id": "company_uuid", # 회사 ID
    "role": "admin",              # 역할
    "type": "access",
    "exp": 1234567890
}

# Refresh Token (7일 유효)
{
    "sub": "user_uuid",
    "type": "refresh",
    "exp": 1234567890
}

```
